# PR Review Assistant

GitHub PRのURLから差分を取得し、レビューを加速するMarkdownレポートを生成する。

## 前提条件

- `gh` CLI が認証済みであること（`gh auth status` で確認）
- 対象リポジトリへの読み取り権限があること

## ワークフロー

### Step 1: PR URLの解析とdiff取得

ユーザーからPR URLを受け取ったら、owner/repoとPR番号を抽出して以下を実行する。
URLが未提供なら「GitHub PRのURLを教えてください」と聞く。

```bash
# メタ情報の取得
gh pr view <PR_NUMBER> --repo <OWNER/REPO> \
  --json title,body,baseRefName,headRefName,files,additions,deletions,changedFiles,labels,author

# diffの取得
gh pr diff <PR_NUMBER> --repo <OWNER/REPO>
```

diffが巨大（3000行超）な場合は `--patch` ではなく `gh api` でファイル単位に分割取得する。

```bash
# ファイル一覧の取得
gh api repos/<OWNER>/<REPO>/pulls/<PR_NUMBER>/files --paginate --jq '.[].filename'

# 個別ファイルのdiff取得（必要に応じて）
gh api repos/<OWNER>/<REPO>/pulls/<PR_NUMBER>/files --paginate \
  --jq '.[] | select(.filename == "TARGET_FILE") | .patch'
```

### Step 2: diff解析と分類

取得したdiffを以下の観点で分類する。

**変更ファイルのカテゴリ分け:**
- Model / DB層（マイグレーション、モデル定義、スキーマ）
- Controller / API層（エンドポイント、リクエスト処理）
- Service / Business Logic層（ユースケース、ドメインロジック）
- View / Frontend層（コンポーネント、テンプレート、CSS）
- Config / Infra層（設定、CI/CD、Dockerfile）
- Test（テストファイル）

**変更の性質の判定:**
- 新機能追加 / バグ修正 / リファクタリング / パフォーマンス改善 / 設定変更

**読み順の優先度（大規模PR時）:**
1. マイグレーション / スキーマ変更（DB構造の把握が最優先）
2. モデル層（リレーション、バリデーション）
3. サービス / ビジネスロジック層
4. コントローラー / API層
5. フロントエンド層
6. テスト
7. 設定 / インフラ

### Step 3: Markdownレポート生成

以下のテンプレートに従い、作業ディレクトリにMarkdownファイルを出力する。
ファイル名: `tmp/pr-review-<PR番号>.md`

## セクション別生成ルール

### 前提知識（セクション2）について
このセクションが本スキルの核心の1つ。レビュアーがdiffを開く前に「何を知っていればこのPRを理解できるか」を網羅する。

**列挙の方針:**
- diff内に実際に登場する全てのフレームワーク固有メソッド、クラス、設定キー、パターンを対象にする
- 「知っていて当然」は想定しない。レビュアーがジュニア〜ミドルでも理解できる粒度で列挙する
- 各解説は1行（40〜60文字程度）に収める。詳しすぎず、でも「それが何か」が分かる粒度
- カテゴリは該当するものだけ出力する（空カテゴリは省略）

**カテゴリ別のピックアップ基準:**
- 言語・フレームワーク基礎: Railsのconcern, callback, scope, TypeScriptのジェネリクス, Vueのcomposition API等
- ライブラリ・Gem: diff内のrequire/importから特定。sidekiq, devise, axios, pinia等
- デザインパターン: Service Object, Form Object, Decorator, Observer, Pub/Sub等
- DB: マイグレーション操作、インデックス種別、外部キー制約、JSONB、ポリモーフィック関連等
- インフラ: Docker命令、CI設定キー、環境変数、デプロイ設定等
- ビジネスドメイン: PRのコンテキストから推測される業務用語（決済、在庫、認証フロー等）

**除外するもの:**
- `if`, `return`, `class`, `def` 等の言語の基本構文
- 広く知られた標準ライブラリの基本メソッド（`map`, `filter`, `push`等）

### シーケンス図（セクション3）について
- API呼び出しやデータの流れが追跡できる変更の場合のみ生成する
- 設定ファイルのみ、CSS変更、テストのみの変更では省略する
- 複数の独立した処理フローがある場合は、それぞれ別のシーケンス図を作成する
- Mermaid sequenceDiagram記法を使用する

### レビューチェック（セクション4）について
- チェック項目は全て記載し、差分に該当しない項目には「該当なし」と注記する
- 問題を発見した場合はファイルパスと行番号（可能な場合）を添える
- 確信度が低い指摘は「要確認」と明記し、偽陽性を減らす
- 重大度: Critical（修正必須） > Warning（修正推奨） > Info（検討余地あり）
