# Code Insight — コミット前の設計意図キュレーション

## このスキルの目的

AIが生成したコード、または自分が書いたコードの設計意図を、
コミット前に「比較・思想・面白い仮説」の3点セットで解説する。

コードは結論であって着火点ではない。
結論だけ見ても「へー」で終わる。
思想→設計判断→実装の順番で提示することで「なぜ？」が自然に湧く構造を作る。

## 実行手順

### 1. diffの取得

まずステージング済みのdiffを確認する：

```bash
git diff --cached
```

ステージングが空の場合はワーキングツリーのdiffにフォールバック：

```bash
git diff
```

両方空の場合は「diffがありません。`git add` してから再実行してください」と伝えて終了。

### 2. 注目ポイントの抽出

diff全体を読み、以下の観点で注目すべきポイントを抽出する：

- 特定のメソッド、クラス、パターンの選択
- フレームワーク固有の規約・慣習の利用
- データ構造やアーキテクチャの判断
- エラーハンドリングやバリデーションの設計
- パフォーマンスやセキュリティに関わる選択

「知ってて当然」レベルの些末な変更（インデント修正、typo修正など）はスキップしてよい。
ただし判断に迷ったら含める側に倒す。

### 3. 各ポイントについて3点セットで解説

各注目ポイントについて、以下の3つを**フルで**提示する。
見出しだけ、一言だけにしない。中身を読んで初めて興味が着火する。

#### 比較（A vs B）
「このコードではAを使っている。Bという選択肢もある。」
- 具体的な代替手段を示す
- それぞれの特徴を簡潔に対比する

#### 思想（なぜこれが存在するのか）
「このメソッド/パターンが生まれた背景にはこういう設計思想がある。」
- フレームワークやライブラリの設計者が何を解決しようとしたのか
- どういう哲学・原則に基づいているのか

#### 面白い仮説（着火用の問い）
「もしかすると、この設計は○○という問題を解決するために生まれたのかもしれない。」
- 断定ではなく仮説として投げる
- 「なぜ？」「本当に？」と掘りたくなるような問いかけ
- 意外性や「そう考えるんだ！」感があるとなお良い

### 4. 出力フォーマット

```
## Code Insight: [ファイル名やコンテキスト]

---

### 1. [ポイントのタイトル]

**比較**: [A] vs [B]
[対比の説明]

**思想**: [背景の設計思想]
[なぜこれが存在するのかの説明]

**💡 面白い仮説**: [着火用の問い]
[仮説の説明]

---

### 2. [次のポイント]
...

---

気になるポイントがあれば、番号を教えてください。そのまま深掘りしましょう。
```

### 5. 出力後の対話

解説を出力した後、ユーザーが番号やキーワードで選んだポイントについて
そのまま対話で深掘りする。ここからは壁打ちモード：

- ユーザーの「なぜ？」に対して一段深い情報を返す
- 比較対象をさらに広げる（A vs B だけでなく C, D も）
- 実際のソースコードやドキュメントを参照して根拠を示す
- ユーザーが自分の言葉で理解を言い直すのを待つ

## 対象技術スタック

以下の技術スタックを中心に解説する。
ユーザーのプロジェクトに応じて適宜追加・更新すること。

- Ruby on Rails
- Vue.js
- TypeScript
- Hotwire（Turbo, Stimulus）

他の技術が含まれている場合も同じ3点セットの構造で対応する。

## やってはいけないこと

- 結論だけ出して終わる（「○○を使っています」だけはNG）
- 見出しやタイトルだけで中身を省略する
- 「こう書くべき」とアドバイスする（問いで返す）
- 些末なポイントで水増しする（本当に面白いものだけ）
- ユーザーが知ってることを長々説明する（知ってたらスキップされるので気にしない）
